## Scope (Hardgrenzen)

**Erlaubt:**
- Alles unter `proto/hermes-eval/` (neuer Pfad)
- Lokaler Test mit einem Modell-Endpoint (OpenRouter / lokales LLM)
- Ein-bis-zwei UFT-typische Tasks als Benchmark-Vergleichspunkt:
  - „Audit den letzten 5-Commit-Diff auf neue Lazy-Stubs" (proxy für `stub-eliminator`)
  - „Scaffold ein neues Format-Plugin für Format X" (proxy für `uft-format-plugin` Skill)
- Vergleich gegen Claude-Code-Lauf desselben Tasks (Wall-Clock + Output-Qualität)
- README in `proto/hermes-eval/README.md` mit gefundenem Ergebnis

**Verboten:**
- Jede Datei außerhalb `proto/hermes-eval/`
- Jede Änderung an `.claude/` Setup
- Jede Migration von echten Agenten/Skills
- Neue Production-Dependency im UFT-Build
- `pip install` ins Haupt-`requirements.txt` oder `pyproject.toml`

**Implicit:** wenn der PoC mehr als 1 Datei außerhalb `proto/` braucht → Annahme
falsch → STOP, das ist eine Integration, kein PoC mehr.

## Verfahren (rigid)

1. **Setup** (½ Tag):
   - `git checkout -b proto/hermes-agent-eval` von main (nach Tag)
   - `git clone https://github.com/NousResearch/hermes-agent.git proto/hermes-eval/upstream`
   - venv unter `proto/hermes-eval/.venv`, dokumentierte deps
   - Modell-Wahl + API-Key in `proto/hermes-eval/.env.example` (echte
     Keys NICHT committen — siehe `.gitignore`-Regel)

2. **Benchmark-Tasks** (3–5 Tage):
   - Task A: Audit der letzten 5 Commits eines disposable Test-Branches
   - Task B: Scaffold ein Test-Format-Plugin (z.B. "tap" — bewusst kein
     reales Format um Verwirrung zu vermeiden)
   - Für jeden Task: identischer Prompt an Claude Code UND hermes-agent
   - Messen: Wall-Clock-Zeit, # Tool-Calls, Output-Qualität (manuelle
     Wertung 1–5), wie viele Rückfragen

3. **Failure-Modes prüfen** (2 Tage):
   - hermes-agent kann ARCH-7 (Teensy VID/PID Disambiguation) korrekt
     erklären, OHNE im Codebase zu suchen?
   - hermes-agent erkennt einen `if (!feature) return DEFAULT;`-Stub als
     Anti-Pattern (per v3-Vollständigkeits-Regel)?
   - hermes-agent versteht den Unterschied zwischen `honest-stub` und
     `lazy-stub` (6 Spalten in `stub-eliminator.md`)?

4. **Cost-Analyse** (½ Tag):
   - Token-Cost pro Task (Claude vs gewähltes hermes-Modell)
   - Migrations-Aufwand: re-encoding von 21 Agenten + 20 Skills + 6
     `CLAUDE.md`-Sektionen in hermes-Format — geschätzt in Personen-Tagen

5. **Bericht** (1 Tag):
   - `proto/hermes-eval/RESULTS.md` mit GO/NO-GO und konkretem Begründungs-
     Datenpunkt (nicht „fühlt sich gut an")
   - GO-Schwelle: hermes ist mindestens **2× besser** in entweder
     Wall-Clock ODER Output-Qualität bei vergleichbarer Korrektheit
   - NO-GO-Schwelle: hermes verfehlt mindestens einen der 3 Failure-Mode-
     Tests (Schritt 3) oder ist > 30% schlechter in Wall-Clock
   - Mittlere Werte zwischen GO/NO-GO → NO-GO (sunk-cost-Schwelle pro
     v3-Anti-Goals: keine neuen Dependencies ohne klaren Gewinn)
