# Recovery Integrity Audit — 2026-04-25

Six bugs found in `src/recovery/` at commit `70e60fc` (UFT v4.1.3). Each
maps to one of the five integrity invariants in `SKILL.md`. They are
documented here as case studies — when reviewing PRs in `src/recovery/`,
check these patterns first.

## I1 — Bytes must be marked unknown if not from media

### Bug 1: `reconstruct_sector` linear interpolation

**Location:** `src/recovery/uft_sector_recovery.c:154-166`

```c
if (gap_len <= 4) {
    uint8_t start_val = (gap_start > 0) ? output[gap_start - 1] : 0;
    uint8_t end_val = output[i];
    for (size_t j = 0; j < gap_len; j++) {
        output[gap_start + j] = start_val +
            (end_val - start_val) * (j + 1) / (gap_len + 1);
    }
}
```

**Why it's wrong:** Disk bytes are not a continuous signal. Treating
`(0xF3, ?, ?, 0x12)` as interpolatable to `(0xF3, 0xAC, 0x65, 0x12)`
fabricates data. There is no `confidence_map` update, so the caller
sees the fabricated bytes as having full confidence.

**Fix pattern:** see SKILL.md Step 3, I1.

## I2 — Probabilistic corrections must verify uniqueness

### Bug 2: `try_single_bit_correction` accepts first match

**Location:** `src/recovery/uft_bitstream_recovery.c:101-122`

```c
for (size_t i = 0; i < len; i++) {
    for (int b = 0; b < 8; b++) {
        data[i] ^= (1 << b);
        if (crc16_ccitt(data, len) == expected_crc) {
            *corrected_byte = i; *corrected_bit = b;
            return 1;          /* FIRST hit accepted */
        }
        data[i] ^= (1 << b);
    }
}
```

**Why it's wrong:** CRC-16 has 65 536 possible values. A 512-byte sector
has 4 096 candidate single-bit flips. Pure-chance collision rate ≈
6 % per sector. The "correction" silently corrupts data in those cases.

**Severity:** Critical for forensic preservation. The downstream
sector_state will report "bad → recovered (1 bit)" even when the
correction was a coincidence.

**Fix pattern:** see SKILL.md Step 3, I2.

## I3 — Verifiability information must propagate

### Bug 3: `vote_byte` ignores `crc_ok`

**Location:** `src/recovery/uft_multiread_pipeline.c:200-247`

```c
static uint8_t vote_byte(const multiread_pass_t *passes, uint8_t pass_count,
                         size_t offset, ...) {
    uint8_t counts[256] = {0};
    for (uint8_t p = 0; p < pass_count; p++) {
        if (offset < passes[p].data_len && passes[p].data) {
            counts[passes[p].data[offset]]++;
            /* passes[p].crc_ok is never consulted */
        }
    }
    /* ... pick max ... */
}
```

**Why it's wrong:** `multiread_pass_t` carries `crc_ok` and `quality`
fields explicitly because these distinguish reads we trust from reads
we don't. Pure majority voting ignores the difference. With 4 of 5
reads CRC-failed, four corrupted reads outvote one good read.

**Fix pattern:** see SKILL.md Step 3, I3.

## I4 — Counter widths must match input scale

### Bug 4: `uint8_t counts[256]` in `average_sector_reads`

**Location:** `src/recovery/uft_sector_recovery.c:95`

```c
uint8_t counts[256] = {0};
for (size_t r = 0; r < read_count; r++) {
    if (reads[r]) counts[reads[r][i]]++;   /* wraps at 256 */
}
```

**Why it's wrong:** `read_count` is `size_t`. With ≥ 256 identical
reads at the same byte position, the counter wraps to 0. Currently
masked because `MULTIREAD_MAX_PASSES = 16`, but the function accepts
any `size_t`. A future caller (stress test, external recovery wizard)
will trip it.

**Fix pattern:** see SKILL.md Step 3, I4.

### Bug 5: same pattern in `vote_byte`

**Location:** `src/recovery/uft_multiread_pipeline.c:209`

Same issue. Currently safe due to MAX_PASSES, but fragile.

## I5 — Confidence is reported, never inferred

### Bug 6: PLL phase update written but never read

**Location:** `src/flux/uft_flux_decoder.c:194-210`

```c
if (pll->use_pll) {
    double error = delta_ns - expected;
    pll->phase  += error * pll->phase_gain;   /* dead write */
    pll->period += error * pll->freq_gain / num_cells;
}
```

**Why it's wrong:** Strictly this isn't an I5 violation — it's a
half-implemented PI controller. Including it here because the failure
mode is the same family: a `confidence_map`-style signal (the phase
correction) is generated but never consumed downstream. PLL lock
quality silently degrades.

**Fix pattern:** apply the phase correction by shifting `prev_time` for
the next iteration, then leak the accumulator. See the audit report.

## How to use this document

When you open a PR touching `src/recovery/`:

1. Check whether your change touches one of the patterns above.
2. If yes, apply the fix pattern from SKILL.md.
3. Annotate any genuinely-intentional violation with
   `/* INTEGRITY-OK: <reason> */` on the relevant line.
4. Run `bash scripts/lint_recovery.sh src/recovery/<your_file>.c` and
   ensure clean exit.

When reviewing someone else's PR, cite these case studies by number
("this looks like Bug 3 — voting without crc_ok") rather than
re-explaining from scratch.
